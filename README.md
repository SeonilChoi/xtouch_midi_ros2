# xtouch_midi_ros2

ROS 2 bridge for the Behringer X-Touch Extender and `ros2_motor_manager`.

The `xtouch_midi` node reads X-Touch MIDI input and publishes the current
8-channel surface state as `midi_msgs/msg/Midi` on `/xtouch/midi`.
MIDI input updates the node's in-memory snapshot, and the node publishes that
snapshot periodically every 5 ms by default.
The `midi_motion_control` node subscribes to `/xtouch/midi`, maps faders to
motor indices, and publishes `motor_status_msgs/msg/MotorStatus` commands on
`/motor_command`.

Published and consumed topics:

| Topic | Type | Payload |
| --- | --- | --- |
| `/xtouch/midi` | `midi_msgs/msg/Midi` | Full 8-channel button, touch, fader, and dial state |
| `/motor_command` | `motor_status_msgs/msg/MotorStatus` | Motor command generated by `midi_motion_control` |

Buttons are latched: pressing once toggles the channel state, and the node sends
MIDI feedback to keep the button LEDs in sync. `btn0[ch]` is ignored. `btn1[ch]`
selects bank offset `+8`, and `btn2[ch]` selects bank offset `+16`. They combine
as bits, so with no bank button active the offset is `+0`, with both active the
offset is `+24`, and the eight physical channels can address motor indices 0
through 31. `btn3[ch]` is the motor-input active gate; it is published true only
when the button is on and that fader has reported at least one update.
`dial[ch]` controls smoothing: `0` is immediate, and `127` uses the full
`max_smoothing_time_ms` time constant. The default is 3000 ms with a squared dial
curve so high dial values produce a much stronger smoothing effect.

The node also sends MCU SysEx display feedback by default. Each channel display
shows `DIALn` on the top row and the current `dial[n]` value on the bottom row.
The default display device id is `21` for X-Touch Extender; use
`display_device_id:=20` if the display does not update on a main X-Touch.

## Install Dependencies

This package uses the MIDI dependency files from `ros2_midi/requirements`.
The paths below assume this workspace layout:

```text
<ws>/src/ros2/xtouch_midi_ros2
<ws>/src/ros2/ros2_midi
<ws>/src/ros2/motion_system_ros2
```

For this machine, `<ws>` is `/home/mini0/colcon_ws`.

Run this once from the ROS 2 source folder:

```bash
cd <ws>/src/ros2
bash ros2_midi/requirements/install_requirements.sh
```

The script installs the apt packages listed in
`ros2_midi/requirements/apt_packages.txt`, including `librtmidi-dev` and
`libasound2-dev`. ROS 2 Humble must already be installed.

Add your user to the `audio` group so the node can access ALSA MIDI devices
without `sudo`:

```bash
sudo usermod -aG audio $USER
```

Log out and back in after changing the group membership. To verify it:

```bash
groups
```

The output should include `audio`.

Check that the X-Touch is visible to ALSA:

```bash
amidi -l
```

The output should include an X-Touch or Behringer MIDI port.

## Build

From the workspace root:

```bash
cd <ws>
source /opt/ros/humble/setup.bash
colcon build --symlink-install \
  --base-paths src/common src/lib src/ros2/motion_system_ros2 src/ros2/xtouch_midi_ros2 \
  --packages-up-to ros2_motor_manager xtouch_midi midi_motion_control
source install/setup.bash
```

The `--base-paths` argument intentionally excludes `src/ros2/ros2_midi/src/xtouch_midi`,
because that older reference package has the same package name: `xtouch_midi`.

## Run

Start the motor manager first, or run it in another terminal:

```bash
source /opt/ros/humble/setup.bash
source <ws>/install/setup.bash
ros2 launch ros2_motor_manager motor_manager_node.launch.py
```

Then start the X-Touch MIDI bridge:

```bash
source /opt/ros/humble/setup.bash
source <ws>/install/setup.bash
ros2 launch xtouch_midi xtouch_node.launch.py
```

To change the periodic MIDI state rate:

```bash
ros2 launch xtouch_midi xtouch_node.launch.py publish_period_ms:=2
```

To try the alternate MCU display id:

```bash
ros2 launch xtouch_midi xtouch_node.launch.py display_device_id:=20
```

Start the MIDI-to-motion bridge with the same motor config used by
`ros2_motor_manager`:

```bash
source /opt/ros/humble/setup.bash
source <ws>/install/setup.bash
ros2 launch midi_motion_control midi_motion_control_node.launch.py \
  config_file:=/absolute/path/to/config.yaml
```

If `config_file` is not supplied, the launch file defaults to
`ros2_motor_manager/config/example.yaml`.

## Verify

In another terminal:

```bash
source /opt/ros/humble/setup.bash
source <ws>/install/setup.bash

ros2 topic echo /motor_command
ros2 topic echo /xtouch/midi
```

Toggle `btn1` and/or `btn2` for a channel, move that fader, then toggle `btn3`
to make the selected motor active. `midi_motion_control` maps physical channel
`chN` to motor `N`, `N + 8`, `N + 16`, or `N + 24` depending on those two bank
buttons.

## Troubleshooting

- `No MIDI input port matching ...`: check USB power/cable and run `amidi -l`.
- `Permission denied`: confirm your user is in the `audio` group with `groups`,
  then log out and back in if it was just added.
- No `/motor_command` output: confirm `/xtouch/midi` is updating and the
  selected bank maps to a motor that exists in the loaded config. `btn3` must be
  on after that fader has reported at least one update.
